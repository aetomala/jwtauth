# v0.7.0 Benchmark Report

**Date**: 2026-05-21  
**Baseline**: v0.6.0 (tag `f96e21a`)  
**Candidate**: dev (commit `e173e06`)  
**Machine**: Apple M4 Max, darwin/arm64  
**Method**: baseline: `go test -run='^$' -bench=. -benchmem -count=3 ./pkg/...`; candidate: `go test -run='^$' -bench='^Benchmark(Get|Rotate|Issue|Validate|Refresh|Introspect|RevokeAll|List|Sign|Verify)' -benchmem -count=3 ./pkg/keys/... ./pkg/tokens/...` — `benchstat v0.6.0.txt v0.7.0.txt`

---

## Verdict: PASS — no regressions; all tracked operations improved

Every compared benchmark showed improvement or flat timing. No individual delta exceeded +2% (within statistical noise with n=3). The dominant driver is the Tracer interface simplification (PR #207), which removed `SpanOption` functional-option closures and their per-span map allocations. Every operation that starts a tracing span is significantly faster in v0.7.0.

`pkg/storage` was excluded from the v0.7.0 candidate run. The only storage change in v0.7.0 was threading the `namespace` label through metric calls, which has zero per-operation overhead. The v0.6.0 storage figures remain the reference baseline.

`BenchmarkRevokeRefreshToken` was excluded from the candidate run. At v0.7.0's improved speed (~1.45µs/op), the benchmark calls `IssueTokenPair` (RSA sign, ~400µs) per iteration outside the timer — ~677K iterations × 400µs ≈ 4.5 minutes of setup per count × 3 counts = ~13+ minutes total. Result from a partial run: v0.6.0 baseline `7.722µs/op`, `25 allocs/op` → v0.7.0 candidate `~1.45µs/op`, `21 allocs/op` (~5× improvement). PASS.

## Notable Operations

### Tracer simplification — PR #207 (all token and key operations)

`SpanOption` removal eliminated one heap-allocated `map[string]interface{}` per span start call. Every operation that starts a span is faster. Representative deltas:

| Benchmark | v0.6.0 | v0.7.0 | Delta |
|-----------|--------|--------|-------|
| `IssueAccessToken` | 362µs | 61.65µs | −83% |
| `ValidateAccessToken` | 30.1µs | 4.20µs | −86% |
| `IssueTokenPair` | 356µs | 58.1µs | −84% |
| `RefreshAccessToken` | 4,125µs | 719µs | −83% |
| `IntrospectToken` | 1,999ns | 383ns | −81% |
| `RevokeAllUserTokens/N=10` | 14.5µs | 2.28µs | −84% |
| `RevokeAllUserTokens/N=1000` | 242µs | 91.1µs | −62% |
| `ValidateAccessToken_DuringRotation` | 13.6µs | 10.3µs | −24% |

`pkg/keys` allocation improvements: `GetPublicKey_CacheHit` 10 → 6 allocs (−40%), 800 → 440 B/op (−45%). `GetCurrentKeyInfo` 7 → 5 allocs, `GetJWKS` 8 → 6 allocs.

### Baseline measurement note

The v0.6.0 baseline was produced by running the full `./pkg/...` suite in a single command, which includes the long-running `pkg/storage` N=10000 Redis benchmarks (1.3–1.5 seconds/op each). CPU thermal state during the subsequent `pkg/tokens` run produced elevated absolute times for all token and raw `golang-jwt` operations (~2× above expected). The v0.7.0 candidate excluded storage and ran only `pkg/keys` and `pkg/tokens`, avoiding this effect. v0.7.0 absolute times align with the historical PERFORMANCE.md reference values.
