# ADR-004: Key ID (kid) Validation at the KeyStore Boundary

**Status**: Accepted  
**Date**: 2026-04-21  
**Deciders**: Architecture Team

## Context

The JWT `kid` header identifies which key to use for signature verification. When a
token arrives, the value in `kid` flows from an attacker-controlled header directly
into KeyStore methods such as `LoadKey`. Before this change, `DiskKeyStore` used the
`kid` value to construct a file path (`dir/<kid>.pem`) without any format check.

A crafted token with `kid: "../../../etc/passwd"` (or any path-traversal sequence)
could reach the OS filesystem call with the attacker's chosen path, potentially:

- Reading arbitrary files on the host (information disclosure)
- Causing unexpected store errors that degrade availability

`RedisKeyStore` had the same gap for defense-in-depth: a malformed `kid` would pass
directly into Redis key construction (`ks:pem:<kid>` / `ks:meta:<kid>`), allowing
key-space pollution or scanning for predictable patterns.

## Decision

**Validate `kid` against a UUID v4 format at every KeyStore method boundary.**

All key IDs generated by `Manager` come from `uuid.New().String()`, which always
produces lowercase UUID v4 strings (e.g. `3a2b5c4d-1e2f-4a3b-8c9d-0e1f2a3b4c5d`).
Any value that does not match this format cannot be a legitimate key ID and must be
rejected before reaching the filesystem or Redis.

The regex enforced at the boundary:

```
^[0-9a-f]{8}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{4}-[0-9a-f]{12}$
```

All four KeyStore methods (`Save`, `UpdateMetadata`, `LoadKey`, `Delete`) return
`ErrKeyStoreInvalidKeyID` immediately when the check fails, before any I/O.

## Consequences

**Positive:**
- Path traversal via `kid` is structurally impossible — attacker strings are rejected
  before any filesystem or Redis call is made
- Defense-in-depth for both store backends
- Validation is centralized in `isValidKeyID()` — a single place to update if the ID
  format ever changes
- Metrics and traces still fire for rejected calls so anomalous patterns are visible

**Negative:**
- Any caller that passes a non-UUID key ID (e.g. tests using "save-key") breaks
  immediately — test fixtures must use real UUID strings
- Slight overhead per KeyStore call (regex match on every invocation, negligible)

**Mitigation:**
- Tests updated to use constant UUID values (`testKeyA`, `testKeyB`, `testKeyMissing`)
- Path traversal cases added as explicit test cases to both DiskKeyStore and
  RedisKeyStore suites

## Alternatives Considered

**Alternative 1: Validate in Manager, not in KeyStore**  
Rejected. Manager controls key generation but KeyStore is the storage boundary that
directly touches the filesystem. Defense must live at the point of use, not at the
caller.

**Alternative 2: Sanitize the kid (strip dots, slashes)**  
Rejected. Sanitization is error-prone and still allows malformed IDs into the store.
An allowlist (UUID regex) is strictly safer — only known-good formats pass.

**Alternative 3: Use filepath.Base to strip path separators**  
Rejected. `filepath.Base("../../../etc/passwd")` returns `passwd`, not an error.
This masks the attack rather than detecting it.

## Security Considerations

- The `kid` header is attacker-controlled and must be treated as untrusted input
- UUID format is a tight allowlist — characters outside `[0-9a-f-]` and any
  deviation from the 8-4-4-4-12 structure are rejected
- Path separators (`/`, `\`, `..`) cannot appear in a UUID; they are rejected
  structurally without special-casing
